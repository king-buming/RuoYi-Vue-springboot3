#### 关键业务逻辑

**1. 审核 → 主表状态联动（TbWorkerAuditServiceImpl.syncStatus）**
- 审核记录的 bizType=worker → 回写 tb_worker.audit_status
- 审核记录的 bizType=cert → 回写 tb_worker_cert.audit_status
- 使用 `@Transactional` 保证数据一致性

**2. 人员删除 = 逻辑删除 + 禁用**
```java
w.setDelFlag("2");
w.setStatus("2");  // 同时设为禁用
```

**3. 角色分配 → 资质校验**
- 保存角色时，调用 `validateRoleRequirements()` 检查角色所需资质是否齐全
- 弱约束设计：角色仍可分配，但返回缺失资质的中文提示列表
- 若缺资质，同时将人员 audit_status 退回 "0"（待审核）

**4. 归档拦截**
- isWorkerActive() 前置校验：delFlag = "0" 且 status != "2" 才允许操作
- 应用于审核、打卡、角色分配等关键操作

**5. 移动端 Token 鉴权（AppTokenUtil）**
- HMAC-SHA256 签名，payload = workerId:expireTime
- 7天有效期，支持 Authorization Header 和 ?token= 参数两种方式
- 不依赖 Spring Security Session，独立鉴权体系

**6. 打卡规则校验（AppCheckinController）**
- 签到：不允许重复签到
- 签退：必须先签到才能签退，不允许重复签退
- 支持 GPS（latitude/longitude）和现场照片（photo_url）
- 归档/禁用人员禁止打卡

## 六、架构设计要点总结

1. **角色设计**：采用"规则表 + 多对多关联"，而非简单枚举。每个角色可独立配置打卡规则（签到/签退/点到）和资质要求，灵活度高。

2. **审核联动**：审核表不独立存在——新增/修改审核记录时自动同步主表状态，事务保证一致性。

3. **逻辑删除**：人员档案采用软删除（delFlag=2 + status=2），保留历史数据。

4. **弱约束资质校验**：角色分配时校验资质，缺资质时标记为"待审核"并返回提示，但不阻断分配流程。

5. **移动端独立鉴权**：`/app/**` 绕过 Spring Security JWT，使用 HMAC-SHA256 轻量 Token 方案。

6. **AI 字段预留**：打卡表的 ai_result、人脸表的 face_feature 已预留字段，面向后续 YOLO / 人脸识别阶段。
